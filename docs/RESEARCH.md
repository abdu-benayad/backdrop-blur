# Native Backdrop Blur for Rust GUI Toolkits: A Research Report

## Executive Answer

Build native backdrop blur by **owning the render loop and inserting one offscreen intermediate texture**: render the UI scene into a texture created with `TEXTURE_BINDING | RENDER_ATTACHMENT`, then run a blur pass that samples that texture in the region behind each frosted surface, and composite the blurred-tinted result masked to a rounded rect — only the final pass writes the swapchain, which is never sampled. This is the same *grab → blur → composite* pipeline that CSS `backdrop-filter` mandates in the W3C spec, that every browser engine implements, that Bevy's bloom node already runs in production wgpu, and that the prior egui glow grab-pass proved on real hardware. The constraint that blocked the earlier egui-*component* design was **not owning the render loop** (a callback-only widget has nothing sampleable behind it because the toolkit draws the whole frame in one clearing pass to an unsampleable surface) — not any inherent limitation of wgpu, glow, or the GPU.

For the project's actual host (an egui library) the "own the loop" verdict is now **concrete and bounded**: it does **not** mean forking egui or eframe. The render-to-intermediate-target capability already exists as a first-class public API one layer below eframe — `egui-wgpu`'s `Renderer::render(render_pass, paint_jobs, screen_descriptor)` accepts a render pass whose color attachment is entirely the caller's choice, and the whole UI flows through that one call ([egui-wgpu renderer.rs](https://github.com/emilk/egui/blob/main/crates/egui-wgpu/src/renderer.rs), CONFIRMED high). The cost is using `egui-winit` + `egui-wgpu` directly instead of eframe's painter — a supported ~100–150-line integration loop, not a hack. eframe itself blocks the clean route: its wgpu painter hardcodes the surface texture as the final color attachment and exposes only `wgpu_render_state()` for in-UI `PaintCallback`s, with no hook to choose the attachment or insert a post-pass ([eframe Frame](https://docs.rs/eframe/latest/eframe/struct.Frame.html), CONFIRMED high).

Once you own the loop, the in-app GPU capability is **clean and reusable** (CONFIRMED high). Whether it is **performant on the actual kiosk GPU is unproven** (PARTIAL) — all blur timings in the literature are desktop or 2015-mobile, not the weak GLES/embedded class a cage kiosk likely runs. The reusable seam is a pure function over textures: `(sampled scene view + source region + blur params {radius, tint, corner-radius mask}) → written into a destination view`. For the algorithm, use a **dual-Kawase (dual-filter) down/up-sample blur** for large or animated radii (the production-compositor standard), falling back to a linear-sampled separable Gaussian for a single small fixed radius. Blur in **linear light** (sRGB texture formats automate decode/encode for opaque content; the premultiplied-alpha edge of a translucent glass panel is a separate concern, §4). For the **cage/wlroots kiosk target there is no OS-side blur** — in-app GPU blur is the only viable route.

A second, host-specific hard part surfaces for immediate-mode egui that the browser model hides: **backdrop staleness**. egui repaints per-viewport, all-or-nothing, reactive by default — there is no region invalidation and no automatic re-snapshot. The Backdrop Root subsystem must own an explicit repaint-driving policy keyed to whether content behind the surface is live (§7), not assume egui will invalidate for it.

## 1. The Problem & What "Backdrop Blur" Requires

"Backdrop blur" (frosted glass / vibrancy / glassmorphism) is a surface whose background is the **blurred, tinted copy of the content currently behind it**. This is categorically different from two things it is routinely confused with:

- **A blurred *shape*** (drop shadow / soft rounded rect). Vello's `draw_blurred_rounded_rect(transform, rect, brush: Color, radius, std_dev)` is exactly this — it synthesizes a blurred shape analytically from geometry + a flat color via Raph Levien's signed-distance-field approximation ([raphlinus.github.io/graphics/2020/04/21/blurred-rounded-rects.html](https://raphlinus.github.io/graphics/2020/04/21/blurred-rounded-rects.html), [docs.rs/vello Scene](https://docs.rs/vello/latest/vello/struct.Scene.html)). It never samples content behind the layer. **This is the single most likely confusion in any survey** and the verification confirmed it (CONFIRMED, high).
- **Whole-window OS vibrancy.** Blurs whatever is behind the *entire window*, not an arbitrary in-app rectangle over opaque app content.

The W3C Filter Effects Level 2 spec defines the real pipeline precisely, and it is the reference a Rust toolkit should implement ([drafts.fxtf.org/filter-effects-2](https://drafts.fxtf.org/filter-effects-2/), CONFIRMED high). For an element `B` with `backdrop-filter`:

1. Copy the **Backdrop Root Image** (content behind `B`, bounded — see below) into a temporary buffer.
2. Apply the filter (blur) to the whole buffer.
3. Apply the inverse of `B`'s transforms.
4. **Clip to `B`'s border box, including `border-radius`.**
5. Draw `B`'s own content over the filtered backdrop.
6. Apply `B`'s effects/clips.
7. Composite into the parent with `source-over`.

This is structurally identical to the proven egui glow grab-pass (copy region behind panel → blur → composite masked to rounded rect). The browsers validate that design at scale.

**The load-bearing correctness mechanism is the "Backdrop Root."** A Backdrop Root is created by the document root and any ancestor with `filter`, `opacity<1`, `mask`, `clip-path`, `backdrop-filter`, `mix-blend-mode`, or `will-change` of those — but **not** by `z-index`, transforms, or positioning. Its purpose is to stop the backdrop image from reading content above an ancestor whose `opacity`/`filter` would otherwise be applied twice (`opacity²` instead of `opacity`) ([drafts.csswg.org/filter-effects-2](https://drafts.csswg.org/filter-effects-2/), CONFIRMED high). This is the spec's answer to the **stacked-frosted-surfaces landmine** (blur-of-blur + double tint): a reusable capability must define an analogous "what content is eligible to be blurred behind surface N" boundary, or overlapping glass panels will double-tint.

Five engineering landmines carry over from the prior findings and are confirmed:
- **Gamma**: blur in linear light or get dark halos (§4).
- **MSAA**: a multisampled read framebuffer breaks `glCopyTexImage2D` (§3).
- **Staleness**: the grab is stale over animating content unless the surface's layer repaints. Browsers avoid this by re-snapshotting every composited frame; an *immediate-mode* toolkit like egui has no automatic re-snapshot and no region invalidation, so the subsystem must drive its own repaints (§7 designs this; it is the host-specific hard part).
- **Stacking**: ordered per-surface passes, each reading the composited-so-far buffer (the Backdrop Root rule).
- **Expense**: every engine documents `backdrop-filter` as forcing an extra render surface; WebKit and Chromium both tell developers to use it sparingly ([webkit.org/blog/3632](https://webkit.org/blog/3632/introducing-backdrop-filters/), CONFIRMED high).

## 2. The Two Fundamental Routes

### Route A — In-app GPU blur (grab-pass / render-to-texture + convolution)

Render UI (or the content behind the frosted surface) into a sampleable offscreen target, blur the region, composite tinted and rounded-rect-masked. **Works on every backend you control, gives per-surface arbitrary-rectangle glass, and is the only route that frosts one in-app layer over another.** Requires owning the render loop (or an engine that exposes a sampleable intermediate). Every browser engine implements backdrop-filter exactly this way — Chromium plumbs it through `cc::EffectNode` → `viz::RenderPass` with a texture readback; Firefox/WebRender (a Rust GPU compositor, the closest analog) uses render tasks ([github.com/mfreed7/backdrop-filter-feature](https://github.com/mfreed7/backdrop-filter-feature), [webkit.org/blog/3632](https://webkit.org/blog/3632/introducing-backdrop-filters/), CONFIRMED high).

### Route B — OS / compositor-level vibrancy

Delegate to the platform compositor, which blurs what is behind the *window*. Route B is a **desktop nicety, not a per-surface capability, and is materially weaker than the draft assumed on both Windows and macOS** — see the caveats below.

- **macOS**: `NSVisualEffectView` (material + `behindWindow` blending), exposed via `window-vibrancy::apply_vibrancy` ([developer.apple.com NSVisualEffectView](https://developer.apple.com/documentation/appkit/nsvisualeffectview)). **macOS Tahoe 26 caveat (the report's OS as of June 2026):** the *behind-window backdrop-blur primitive itself survives* — `underWindowBackground` is still the only public path to true desktop-backdrop blur, and `window-vibrancy` 0.7.1 (Nov 2025) still exposes `apply_vibrancy/apply_blur` over `NSVisualEffectView` unchanged ([lib.rs/window-vibrancy](https://lib.rs/crates/window-vibrancy), CONFIRMED high). What *is* superseded is the **legacy named-material sidebar idiom**: WWDC25 session 310 states verbatim that an `NSVisualEffectView` showing legacy sidebar material on Tahoe "will prevent the glass material from showing through" and "should be removed," because `NSSplitViewController` now applies sidebar/inspector glass automatically ([developer.apple.com/videos/play/wwdc2025/310](https://developer.apple.com/videos/play/wwdc2025/310/), CONFIRMED high). **Liquid Glass is a distinct capability, not a backdrop-blur replacement:** `NSGlassEffectView` (public AppKit, macOS 26.0+) *wraps your in-app content* in glass via its `contentView` and Apple explicitly warns against placing it behind content as a sibling — it uses lensing/refraction, not scatter-blur, and does not grab what is behind the window ([learn.microsoft.com AppKit NSGlassEffectView](https://learn.microsoft.com/en-us/dotnet/api/appkit.nsglasseffectview?view=net-macos-26.0-10.0), [developer.apple.com/documentation/appkit/nsglasseffectview](https://developer.apple.com/documentation/appkit/nsglasseffectview), high). In the Rust ecosystem the `NSGlassEffectView` path is only a fork/PR in `window-vibrancy` (#198/#199, buggy window-edge misalignment) or the separate `tauri-plugin-liquid-glass`, whose own README treats `NSGlassEffectView` as a *private* API with App-Store-review risk and falls back to `NSVisualEffectView` `UnderWindowBackground` on macOS <26 ([github.com/hkandala/tauri-plugin-liquid-glass](https://github.com/hkandala/tauri-plugin-liquid-glass), [window-vibrancy#198](https://github.com/tauri-apps/window-vibrancy/issues/198), medium). **Net macOS caveat:** keep `underWindowBackground` as the live behind-window blur route; drop the named decorative `NSVisualEffectMaterial` variants from any recommendation; treat Liquid Glass as content-glass, not backdrop.
- **Windows 11**: DWM `DwmSetWindowAttribute(DWMWA_SYSTEMBACKDROP_TYPE=38)` → Mica (static, once-sampled wallpaper, cheap), **Acrylic** (real-time frosted glass, transient surfaces, GPU-intensive, auto-disabled under Battery Saver), Tabbed; via `window-vibrancy::apply_acrylic/apply_mica` ([learn.microsoft.com/.../ne-dwmapi-dwm_systembackdrop_type](https://learn.microsoft.com/en-us/windows/win32/api/dwmapi/ne-dwmapi-dwm_systembackdrop_type), CONFIRMED high *for native Win32 apps*). **Critical wgpu caveat — likely non-functional for an egui/wgpu app (DISPUTED → likely-blocked, see §3).** Mica/Acrylic only render where the client area presents nothing opaque (the Tauri idiom is a zero-alpha `#00000000` background, not a transparent-OS-window flag). wgpu's default DX12 swapchain is created directly from the HWND (`Dx12SwapchainKind::DxgiFromHwnd`), whose own rustdoc states verbatim it "does not support transparency"; only the DirectComposition variant (`DxgiFromVisual`) "supports transparent windows" ([docs.rs/wgpu Dx12SwapchainKind](https://docs.rs/wgpu/latest/wgpu/enum.Dx12SwapchainKind.html)). wgpu PR #7117 (merged 2025-02-12) confirms the rule at the API level — "Regular swapchains should only have Opaque"; `PreMultiplied`/`PostMultiplied` are conditionally available *only* on DirectComposition surfaces ([gfx-rs/wgpu#7117](https://github.com/gfx-rs/wgpu/pull/7117)). So an opaque wgpu present over the full client area occludes the DWM backdrop. eframe's own transparency story compounds this: the Mac case (#2680) was fixed, but Windows transparency is still broken (#4451 "renders black instead of transparent", open) and the acrylic/mica request (#3050) has no shipped wgpu solution ([emilk/egui#4451](https://github.com/emilk/egui/issues/4451), [emilk/egui#3050](https://github.com/emilk/egui/issues/3050)). Tauri "works" because its UI is DWM-composited WebView2, not a wgpu swapchain owning the client area — it is **not** evidence an egui/wgpu app gets the same result. A non-default escape hatch exists (a `DxgiFromVisual` DirectComposition swapchain), but eframe does not wire it up, and wgpu's DirectComposition transparency had a live alpha bug as of 2025 ([gfx-rs/wgpu#7108](https://github.com/gfx-rs/wgpu/issues/7108)).
- **Wayland**: KDE's `org_kde_kwin_blur` (KWin-specific, since 2015) and the new standardized `ext-background-effect-v1` (merged to wayland-protocols staging ~2026; client calls `set_blur_region`, compositor blurs). Implemented by KWin/Plasma 6.7, niri 26.04, COSMIC — **not by wlroots/Sway** (upstream explicitly refuses blur) ([wayland.app/protocols/ext-background-effect-v1](https://wayland.app/protocols/ext-background-effect-v1), [phoronix.com/news/Wayland-Background-Effect](https://www.phoronix.com/news/Wayland-Background-Effect), CONFIRMED high).
- **X11**: picom (user-config dual_kawase/kawase/gaussian), not app-controllable ([man.archlinux.org/man/picom.1](https://man.archlinux.org/man/picom.1), medium).

**When each applies.** Route B is the cheapest *correct* path when (a) you want one frosted *window* over the desktop, (b) the platform supports it for your rendering topology, and (c) the surface is partially transparent with actual backdrop content behind it. It **cannot** produce per-surface glass over opaque in-app content — Microsoft's own docs separate "system backdrops" (whole-window) from "in-app acrylic" (a WinUI Composition feature not exposed to an egui/wgpu app) ([learn.microsoft.com/.../system-backdrops](https://learn.microsoft.com/en-us/windows/apps/develop/ui/system-backdrops), CONFIRMED high). `window-vibrancy` is **whole-window-only, raw-handle-based, and does nothing on Linux** ("Linux: Unsupported, Blur and any vibrancy effects are controlled by the compositor", [docs.rs/window-vibrancy](https://docs.rs/window-vibrancy/latest/window_vibrancy/), CONFIRMED high). Combined with the Windows-wgpu occlusion problem and the macOS named-material churn, Route B is a *fragile* desktop nicety; **a reusable per-rectangle frosted-glass capability must be Route A.**

## 3. Backend Reachability Matrix

| Backend | In-app backdrop blur reachable? | Mechanism | Key constraint |
|---|---|---|---|
| **wgpu (own the loop)** | **Yes — clean** | Offscreen `TEXTURE_BINDING\|RENDER_ATTACHMENT` target → blur pass → composite; final pass → swapchain. For egui: `egui-wgpu::Renderer::render` into a caller-chosen attachment | Must insert the intermediate target; impossible from a callback-only widget or via eframe's painter |
| **wgpu (component lib, no loop)** | **No** | — | Surface guarantees only `RENDER_ATTACHMENT`; single clearing pass; no subpass/input-attachment |
| **glow / GL / GLES (own the loop)** | **Yes — proven** | `glCopyTexSubImage2D`/`glBlitFramebuffer` grab → separable/Kawase ping-pong FBO → composite | MSAA read FB breaks the copy; GLES3 needed for blit; gamma |
| **glow via egui `PaintCallback`** | Yes, with hazard | Raw `glow::Context` from `Painter::gl()` inside `CallbackFn` | Version-coupling downcast silently no-ops on type mismatch |
| **Software (CPU)** | Yes (CPU), grab path unverified | 3-pass box-blur Gaussian approximation (Skia's technique) | Pixel-buffer readback path from tiny-skia/egui software backend not verified; no cited CPU frame-time figure (§3, PARTIAL) |
| **Web (wasm)** | Yes — same as native | Blur in WebGL/WebGPU yourself | CSS `backdrop-filter` cannot frost in-canvas layers against each other |

### The wgpu verdict, stated explicitly

**Owning the render loop makes wgpu backdrop blur clean and reusable.** The prior "ruled out for wgpu" finding was a constraint of *not owning the loop*, not of wgpu. Verification confirmed both halves:

- The blocker is precisely that the **wgpu surface texture only guarantees `TextureUsages::RENDER_ATTACHMENT`** — "The only usage guaranteed to be supported is `TextureUsages::RENDER_ATTACHMENT`" ([docs.rs/wgpu SurfaceConfiguration](https://docs.rs/wgpu/latest/wgpu/type.SurfaceConfiguration.html)) — combined with the toolkit drawing the whole frame in one clearing pass. Requesting `COPY_SRC` on a surface broke DX12/Vulkan ([gfx-rs/wgpu#1172](https://github.com/gfx-rs/wgpu/issues/1172)). **This is NOT an inherent wgpu limitation** (CONFIRMED, high). An offscreen texture you create yourself carries `TEXTURE_BINDING | RENDER_ATTACHMENT` freely; the learn-wgpu HDR chapter renders the scene to an `Rgba16Float` intermediate and samples it in a tonemap pass ([sotrh.github.io/learn-wgpu/.../tutorial13-hdr](https://sotrh.github.io/learn-wgpu/intermediate/tutorial13-hdr/)). The egui-specific corroboration: "you can't generally read from the same image texture onto which you're writing; wgpu will panic if you try" — and the fix is a two-texture front/back swap ([mxs.dev/blog/egui-wgpu-blurry-windows](https://www.mxs.dev/blog/egui-wgpu-blurry-windows)).

- **Bevy's bloom is the production existence proof** (CONFIRMED, high — verified against source twice). In `crates/bevy_post_process/src/bloom/mod.rs`, the bloom node reads `view_target.main_texture_view()` — the **offscreen HDR scene texture, not the swapchain** (the "out" texture is written only by the final blit; "the user never sees the main texture", [PR #6415](https://github.com/bevyengine/bevy/pull/6415)) — runs a downsample mip-chain then `LoadOp::Load` + `set_blend_constant` upsample-blend ping-pong, and composites back via `view_target.get_unsampled_color_attachment()`. Critically, the **threshold/prefilter is off by default** (`NATURAL` preset, `threshold: 0.0`; the `USE_THRESHOLD` branch is `#ifdef`-compiled-out when `threshold <= 0`), and Bevy ships a `SCREEN_BLUR` preset that blurs the **entire** frame — so bloom is genuinely "grab whatever was rendered, blur it, composite back," not an emissive-only special case ([bloom.wgsl](https://github.com/bevyengine/bevy/blob/main/crates/bevy_post_process/src/bloom/bloom.wgsl), [settings.rs](https://github.com/bevyengine/bevy/blob/main/crates/bevy_post_process/src/bloom/settings.rs)). The only differences from backdrop blur are masking to a rounded rect and tinting — a compositing-shader change, not a pipeline change.

- **The egui-specific seam is `egui-wgpu::Renderer`, and it does not require forking** (CONFIRMED, high — the linchpin the prior draft only asserted). `Renderer::render(render_pass: &mut wgpu::RenderPass<'static>, paint_jobs: &[ClippedPrimitive], screen_descriptor: &ScreenDescriptor)` "Executes the egui renderer onto an existing wgpu renderpass" — the render pass and therefore its color attachment are created by the *caller*, so it can target an offscreen texture as easily as the swapchain, and the whole UI flows through it because `paint_jobs` is the complete tessellated frame from egui's `Context` ([egui-wgpu renderer.rs](https://github.com/emilk/egui/blob/main/crates/egui-wgpu/src/renderer.rs), [docs.rs/egui-wgpu](https://docs.rs/egui-wgpu)). A real winit+wgpu loop demonstrates the swap concretely: `egui_rpass.execute(&mut encoder, &output_view, &paint_jobs, …)` where `output_view` is just whatever `wgpu::TextureView` the app chose — substitute a view of an app-owned offscreen texture and the entire UI renders there with no other change ([hasenbanck/egui_example](https://github.com/hasenbanck/egui_example)). This makes "own the loop" concrete: the ordinary `egui-winit` + `egui-wgpu` integration (~100–150 lines) plus one post-pass that samples the intermediate and composites to the surface.

- **eframe blocks the clean route.** eframe's wgpu painter calls `begin_render_pass` with the surface texture (`output_frame.texture` / a capture texture) hardcoded as the color attachment, and the only wgpu accessor `eframe::Frame` exposes is `wgpu_render_state()`, documented for in-UI `PaintCallback`s — there is no API to substitute an offscreen target or run a full-frame post-pass ([docs.rs/eframe Frame](https://docs.rs/eframe/latest/eframe/struct.Frame.html), [eframe web_painter_wgpu.rs](https://github.com/emilk/egui/blob/master/crates/eframe/src/web/web_painter_wgpu.rs)). The in-eframe alternative — a localized blur `PaintCallback` — is explicitly a proof-of-concept: its author uses `std::mem::transmute` to nest render passes (which wgpu does not normally permit) and calls it "more of a proof of concept" with no perf numbers ([mxs.dev/blog/egui-wgpu-blurry-windows](https://www.mxs.dev/blog/egui-wgpu-blurry-windows)); the maintainer-blessed `CallbackTrait::prepare`-renders-to-texture pattern samples the callback's *own* texture, not the live frame behind the panel, so it is not true backdrop blur ([emilk/egui#4583](https://github.com/emilk/egui/discussions/4583)). egui#5889 ("Switch to wgpu as default backend of eframe") concerns the default backend, not render-to-texture hooks — so "no upstream eframe plan" holds but is **not load-bearing**, because the capability lives in `egui-wgpu` below eframe ([emilk/egui#5889](https://github.com/emilk/egui/issues/5889)). Dropping a `TextureHandle` in the same frame still panics ([emilk/egui#5224](https://github.com/emilk/egui/issues/5224)), so resource lifetimes inside the loop need care. **No standalone general-purpose wgpu backdrop-blur crate exists** — blur ships only embedded in full renderers (Bevy `bevy_post_process`, `soorat`), so the toolkit must extract this pass itself (medium).

### The glow/GLES path

The grab-pass **generalizes to any glow loop**, not just egui (CONFIRMED, high). egui_glow's `CallbackFn` yields the raw `Arc<glow::Context>` via `Painter::gl()`, but routing through egui's `PaintCallback` carries a documented hazard: egui downcasts the boxed callback to the *exact* `egui_glow::CallbackFn` type the linked version was compiled against, and a mismatch **silently no-ops** ([doc.servo.org/egui_glow/.../CallbackFn](https://doc.servo.org/egui_glow/painter/struct.CallbackFn.html)). A hand-rolled loop (the `pure_glow.rs` pattern) calls the grab-pass GL directly between paint and swap with no downcast and no version coupling ([egui_glow/examples/pure_glow.rs](https://github.com/emilk/egui/blob/main/crates/egui_glow/examples/pure_glow.rs)).

**Grab mechanism ranking** (CONFIRMED, high): `glBlitFramebuffer` (GLES3+, fastest, doubles as MSAA resolve) > `glCopyTexSubImage2D` (GPU-resident, writes pre-allocated storage, stalls with implicit flush) >> `glReadPixels` (CPU readback, avoid). The measured 0.2–0.5 ms for a tooltip-sized panel already absorbs the copy stall.

**The MSAA landmine is real but does NOT bite stock eframe by default — with a critical caveat (PARTIAL).** eframe's `NativeOptions.multisampling` defaults to `0` (MSAA off) — confirmed verbatim in `epi.rs` ([docs.rs/eframe NativeOptions](https://docs.rs/eframe/latest/eframe/struct.NativeOptions.html)). **But** the inference "therefore the default framebuffer is single-sample" is *not* guaranteed: eframe only constrains samples when `multisampling > 0`, otherwise leaves it unspecified on glutin's `ConfigTemplateBuilder` ("By default multisampling is not specified"), and picks the *first* matching EGL config with no `num_samples()` filtering. EGL's sort order puts zero-sample configs first on conformant platforms, so in practice the grab works — but a non-conformant driver, a driver-control-panel "force MSAA" override, or an unusual EGL enumeration can still hand back a multisampled default framebuffer. **The runtime `glGetIntegerv(GL_SAMPLES, ...)` query on the default framebuffer is therefore a mandatory, unconditional guard** (Dolphin does exactly this), not optional — especially on the EGL-driven kiosk path. When `samples > 0`, resolve via `glBlitFramebuffer` into a single-sample FBO before copying.

`EXT_shader_framebuffer_fetch` is **not** a blur-grab alternative (CONFIRMED, high): it exposes only the fragment's own `(x,y,sample)` position, never neighbors, so it cannot feed a convolution ([registry.khronos.org/.../EXT_shader_framebuffer_fetch.txt](https://registry.khronos.org/OpenGL/extensions/EXT/EXT_shader_framebuffer_fetch.txt)). **Caveat (PARTIAL):** the *generalization* "no tiler can read neighbors" is now false — `VK_QCOM_tile_shading` (Vulkan 1.4.312, April 2025) adds an overlapping-tile "apron" allowing reads outside the current tile, explicitly motivated by convolution/blur — but it is Adreno-only, feature-gated, and brand-new, so it does not change the portable recommendation. Framebuffer-fetch's tile-local read *could* still cheapen the final tint/composite step on tilers (open question).

### The software (CPU) path

The CPU backend is **first-class in the research goal (wgpu / glow / software / web) but the grab path is unverified (PARTIAL).** The intended algorithm is the standard 3-pass box-blur Gaussian approximation (Skia's `window = floor(sigma·3·√(2π)/4 + 0.5)`), which is correct and cheap in principle, and box quality is acceptable for a fallback branch. The unverified parts are load-bearing and must be probed before claiming the backend works: (1) **whether egui's software backend / tiny-skia even exposes the rendered pixel buffer for a region grab** — no source confirms a readback path; (2) **the CPU cost at a UI frame budget** — no cited CPU-blur frame-time figure exists, so "acceptable" is an assertion, not a measurement; SIMD box-blur feasibility at tooltip/dialog sizes is plausible but unmeasured; (3) **the gamma round-trip on CPU** is more expensive than the GPU's free sRGB sample/encode and must be done explicitly in >8-bit intermediates to avoid banding. For a cage kiosk that may fall back to software, this backend needs its own de-risk probe (Open Question 5/§8), not a one-line dismissal.

### The web (wasm) path

A Rust UI rendering its whole interface into one `<canvas>` **cannot** use CSS `backdrop-filter` to frost one in-canvas layer over another — `backdrop-filter` operates on the DOM compositing tree, and the canvas is one opaque DOM leaf with no internal boundary ([developer.mozilla.org/.../backdrop-filter](https://developer.mozilla.org/en-US/docs/Web/CSS/backdrop-filter), medium). Two consequences: (a) if the frosted panel is a *separate DOM overlay* above the canvas, CSS `backdrop-filter` is the cheapest correct path (canvas pixels are valid backdrop input); (b) if the panel is drawn *inside* the toolkit's canvas (the normal immediate-mode case), the toolkit must implement the grab-pass in WebGL/WebGPU exactly as on native.

## 4. Blur Algorithm Choice

**Recommendation: dual-Kawase (dual-filter) for large/animated radii; linear-sampled separable Gaussian for a single small fixed radius. Always blur in linear light. Downsample-then-blur is mandatory on bandwidth-bound (tiler/kiosk) GPUs. For the translucent rounded-rect edge, default to premultiplied-alpha linear-space blur (§"Gamma").**

### Dual-Kawase (dual-filter), the production standard

Invented by Marius Bjørge (ARM), SIGGRAPH 2015 "Bandwidth-Efficient Rendering." Downsample N times with a 5-tap kernel, upsample N times with an 8-tap kernel; **radius is controlled by iteration count, not kernel size** (each pass ~doubles effective radius). The **exact tap pattern** (CONFIRMED, high — verified against the ARM slides, picom, and current KWin master, three independent shipping codebases):

- **Downsample (5 taps):** `center×4 + 4 diagonal corners×1, ÷8` (ARM normalized: center 1/2, corners 1/8).
- **Upsample (8 taps):** `4 cardinal×1 + 4 diagonal×2, ÷12` (ARM normalized: cardinals 1/12, diagonals 1/6).

Shipped verbatim in picom's `gl_common.c` ([picom@b698735](https://github.com/yshui/picom/commit/b6987352499bd23b8eae73553e64aa9169bcb6e6)) and KWin's `downsample.frag`/`upsample.frag` on master ([invent.kde.org/plasma/kwin/.../upsample.frag](https://invent.kde.org/plasma/kwin/-/raw/master/src/plugins/blur/shaders/upsample.frag)). The divisors equal the weight sums, so the filter is energy-preserving. **Implementation caveat (low confidence — the specific double-apply failure mode is inferred, not confirmed by a side-by-side diff of both current sources):** picom and KWin use *different offset-encoding conventions* (picom cardinals at distance 1.0; KWin scales by a `halfpixel` uniform). The KWin upsample shader was read directly (`uniform vec2 halfpixel`, offset scale, 4 cardinal ×1 + 4 diagonal ×2, ÷12); picom's current `gl_common.c` offset convention was *not* re-read in verification. Treat the warning as: pick **one** convention and derive offsets consistently; do not paste both verbatim into one codebase without checking whether the half-pixel factor is encoded in the offset or applied separately.

**Performance** (CONFIRMED for the *relative ordering and bandwidth rationale*; the absolute milliseconds do not transfer to 2026 hardware). On a Mali-T760 MP8 (a tiler — the kiosk class) at 1080p for a 97×97-equivalent blur: **Dual 2.8 ms, Kawase 4.5 ms, separable 5×5 Gaussian 23.5 ms, full Gaussian 41.9 ms, box 21.4 ms** ([ARM SIGGRAPH 2015 slides](https://community.arm.com/cfs-file/__key/communityserver-blogs-components-weblogfiles/00-00-00-20-66/siggraph2015_2D00_mmg_2D00_marius_2D00_slides.pdf)). Dual wins on **bandwidth** (5% reads vs 98% for linear-sampled Gaussian — ~7% of total) and **cache** (71% hit rate), which is why it dominates specifically on small-cache mobile/tiler GPUs. The modern corroboration: dual Kawase held ~0.8–0.9 ms flat from radius 20→1000 on an RTX 3080 Ti at 4K while Gaussian scaled 5.5→108 ms ([frost.kiwi/dual-kawase](https://blog.frost.kiwi/dual-kawase/)).

**Honest caveats:** (1) the 2015 milliseconds are single-vendor and do not transfer to 2026 hardware — only the *relative ordering and bandwidth rationale* are durable, and **no 2026-era weak-GLES figure exists for the actual kiosk GPU** (this is why the executive performance verdict is PARTIAL, not CONFIRMED); (2) the advantage requires a **reasonably large radius** (≥7×7); at small radii dual and Gaussian converge, so for a tiny tooltip blur the separable Gaussian is fine; (3) Intel found Kawase "unexpectedly slow on NVIDIA" in one case — a dispatch-tuning hazard, not an algorithmic reversal ([intel.com/.../image-blur-algorithms](https://www.intel.com/content/www/us/en/developer/articles/technical/an-investigation-of-fast-real-time-gpu-based-image-blur-algorithms.html)).

**Quality:** dual Kawase is visually indistinguishable from a true Gaussian for UI/bloom (PSNR 49.78 dB vs Gaussian reference) and is temporally stable under sub-pixel motion, unlike naive mipmap blur which shows blocky aliasing (ARM slides 25–29, CONFIRMED high). Production compositors add **perceptual-space dithered noise** to suppress banding from the reduced-resolution blur (KWin moved its noise into gamma space, [invent.kde.org/.../merge_requests/1281](https://invent.kde.org/plasma/kwin/-/merge_requests/1281)).

### Separable Gaussian (the small-fixed-radius fallback)

A 1D 9-tap Gaussian collapses to 5 fetches via the rastergrid bilinear-sampling trick ([rastergrid.com/.../efficient-gaussian-blur-with-linear-sampling](https://www.rastergrid.com/blog/2010/09/efficient-gaussian-blur-with-linear-sampling/)). On a desktop GTX 1060 at 1080p, a 33-tap separable blur measured **0.54 ms** (with bilinear), vs 1 ms (separable, no bilinear), vs 15 ms (naive single-pass) ([lisyarus.github.io/.../compute-blur.html](https://lisyarus.github.io/blog/posts/compute-blur.html)).

**Performance is a non-barrier for a small panel on a desktop GPU, but this is scoped, not universal (PARTIAL).** Two over-statements must be corrected: (1) **"fragment beats compute" is one desktop-GPU data point** — lisyarus measured fragment 0.54 ms vs best compute 1.06 ms on a GTX 1060, but the author *explicitly disclaims generalization* ("I genuinely have no idea" why compute lost); the ordering is unverified on integrated/GLES hardware. (2) **The sub-ms numbers are desktop, not the cage/embedded kiosk target** — no 2026-era weak-GLES figures were found, and the literature consistently routes weak hardware to *downsampling* (KWin needed dedicated optimization to hit 60 fps on integrated Intel; GPUImage needed 4× downsampling for 30 fps mobile). The escape hatch that keeps the claim true: blur cost scales with blurred-texel count, so a panel is a fraction of full-screen — but a **full-screen or large frosted surface on a kiosk iGPU is a real frame-budget concern**, and the production choice there is downsampled Kawase, not a 33-tap separable Gaussian. The "downsample-then-blur" trick is near-universal: Flutter's Impeller, WebRender, and Skia's GPU path all downsample first so kernel cost stays bounded by radius ([flutter/flutter#140949](https://github.com/flutter/flutter/issues/140949)).

### Gamma correctness — non-negotiable, with a translucency caveat (PARTIAL)

Blur **must** be computed in linear light. Averaging gamma-encoded sRGB values underestimates the true linear average (the sRGB curve is convex), darkest where two bright differently-colored regions meet — the diagnosed dark halo ([NVIDIA GPU Gems 3 Ch.24](https://developer.nvidia.com/gpugems/gpugems3/part-iv-image-effects/chapter-24-importance-being-linear); [iquilezles.org/articles/gamma](https://iquilezles.org/articles/gamma/), CONFIRMED high). Two correct routes: (a) sample → linearize in-shader → blur → re-encode; (b) use sRGB-typed texture/FBO formats so the GPU linearizes on read and encodes on write (`GL_FRAMEBUFFER_SRGB`). **sRGB formats automate this for opaque content** ([learnopengl.com/Advanced-Lighting/Gamma-Correction](https://learnopengl.com/Advanced-Lighting/Gamma-Correction)).

**For a *translucent* glass panel the automation is incomplete, and this is the one correctness case that actually bites the project's rounded-rect frosted panel (PARTIAL).** The concrete failure: at the soft alpha border of a rounded rect, straight-vs-premultiplied alpha produces *visibly different* edge artifacts (dark fringe vs white halo), and sRGB formats do **not** handle it for you. Specifics: (1) **alpha is not gamma-corrected** by sRGB formats (correct — alpha is already linear — but it means the format gives no free pass on the alpha math); (2) **premultiplied-alpha at soft edges is a separate, non-automated step** — decode-to-linear (>8 bit), multiply by linear alpha, re-encode ([hacksoflife premult sRGB](http://hacksoflife.blogspot.com/2022/06/srgb-pre-multiplied-alpha-and.html), [ssp.impulsetrain.com/gamma-premult](https://ssp.impulsetrain.com/gamma-premult.html)); (3) some embedded/mobile GPUs may convert *after* filtering (cheaper, slightly wrong) — relevant to the kiosk target; (4) keep linear intermediates at **>8 bits** (RGBA16F, or an sRGB-format target with higher internal precision) — round-tripping through linear-U8 bands the darks. **Default recommendation (the implementer must pick one, so this report picks): blur in linear light with premultiplied alpha**, matching the compositor convention used by KWin/picom/WebRender and avoiding the dark-fringe artifact at the panel edge; verify against a committed snapshot of a glass panel over a high-contrast edge. The residual tension worth a probe: the W3C spec says backdrop filters operate in *sRGB*, which conflicts with the linear-blur recommendation — open question on whether box-blur+downsample masks the gamma error enough that the spec tolerates it (§8).

## 6. The Kiosk / cage Reality

**Under cage there is no OS-side blur, and in-app GPU blur is the only viable route.** This was the most-disputed claim in the research, and verification **REFUTED** the optimistic reading.

- **`org_kde_kwin_blur` is NOT implemented by cage** (DISPUTED → REFUTED, high). The wayland.app "Cage 0.2.0" listing is a **false positive**: wayland.app auto-generates its tables by mining protocol-XML references across repos, recording *advertised globals*, not *rendered effects*. cage's actual source (`cage.c`, 7 `.c` files total) creates 21 wlroots globals — **none is a blur manager**; a GitHub code search for "blur"/"org_kde_kwin_blur" in the repo returns zero hits; `meson.build` depends only on plain wlroots-0.20 (no scenefx). cage's current release is 0.3 (the table's "0.2.0" is stale). A compositor cannot advertise a global it does not implement, and there is no blur code to inspect ([github.com/cage-kiosk/cage/blob/master/cage.c](https://github.com/cage-kiosk/cage/blob/master/cage.c)). (Corroboration: the same wayland.app table lists Sway/river/Weston, which upstream-*refuse* blur — confirming the table records advertisement, not capability.)

- **Stock wlroots renders no blur.** Upstream wlroots/Sway explicitly refuse it ([swaywm/sway#4356](https://github.com/swaywm/sway/issues/4356)). Blur in the wlroots ecosystem comes only from the third-party **scenefx** drop-in scene-API replacement (MIT, GLESv2, pre-1.0, "not yet ready for other compositors") used by SwayFX/MangoWC — and SwayFX blur is *compositor-side config* (`blur enable`), not client-driven via the KDE protocol ([github.com/wlrfx/scenefx](https://github.com/wlrfx/scenefx)).

- **The new standardized `ext-background-effect-v1` exists in Rust** (`wayland_protocols::ext::background_effect::v1`, ≥0.32.12, client+server; winit added client support) and is implemented by KWin/Plasma 6.7, niri 26.04, COSMIC — **but none is wlroots-based, and cage implements neither protocol** ([smithay.github.io/.../background_effect](https://smithay.github.io/wayland-rs/wayland_protocols/ext/background_effect/index.html)).

- **Even if a protocol were present, it is semantically moot on a kiosk:** a single fullscreen app has nothing behind its window for the compositor to blur. Per-surface glass inside the kiosk app is by definition an in-app effect the compositor cannot see.

- **GLES is a constraint, not a ceiling** (CONFIRMED, high). The egui *client* negotiates its own EGL/GL version via glutin/Mesa, **independent of wlroots' internal GLES2 compositor renderer** — the client↔compositor boundary is a dma-buf handle, not a shared context. A desktop-class Mesa driver readily grants GLES 3.x or desktop GL, enabling `glBlitFramebuffer` and `#version 300 es`. The real ceiling is the *deployment GPU + Mesa driver*, not the compositor, so the design must **probe the context version at runtime** (`glGetString(GL_VERSION)` / EGL config query) and keep a GLES2 fallback (`glCopyTexSubImage2D`, no blit-resolve, forbid MSAA). GLES3 fragment shaders mandate a default precision qualifier; prefer `mediump` where the blur math tolerates it.

**Conclusion for the kiosk:** invest in the in-app GPU grab-pass being robust (gamma-correct, MSAA-guarded, GLES2-fallback-capable, with a live-content repaint policy per §7) — it is the *only* frosted-glass mechanism on a cage kiosk.

## 7. Recommended Architecture for a Reusable Rust Backdrop-Blur Capability

### The public seam (pure function over textures)

The reusable contract is a pure pass with no app coupling:

```
backdrop_blur(
    scene: &SampledTextureView,      // the already-rendered content to blur
    region: BackdropRect,            // source sub-region in framebuffer pixels
    params: BlurParams {             // radius (→ iteration count), tint, corner-radius mask
        iterations, sampling_offset, tint: LinearRgba, corner_radius, alpha,
    },
    out: &DestinationView,           // where the frosted surface is composited
) -> ()
```

Bevy's `ViewTarget::post_process_write()` `{source, destination}` ping-pong is the template for the read-and-write-the-screen-safely problem. The seam is the *same* on wgpu and glow; only the resource types differ (`wgpu::TextureView` vs glow FBO/texture).

### Where it sits relative to the render loop

The capability **requires owning the loop**, and for egui that means **driving `egui-winit` + `egui-wgpu::Renderer` (or `egui-glow`) directly rather than using eframe's painter** (§3). Insert one decision: render UI (or at least the content eligible to be a backdrop) into an **offscreen intermediate** (`TEXTURE_BINDING | RENDER_ATTACHMENT`, a >8-bit linear or sRGB-typed format) — for wgpu, by passing a view of that texture as the color attachment to `Renderer::render` — then for each frosted surface in paint order run the blur pass reading the composited-so-far intermediate, composite tinted + rounded-rect-masked, and only the *final* pass writes the swapchain.

**Stacking discipline (the Backdrop Root rule):** process frosted surfaces in z-order, each reading the buffer *as composited so far* (including prior frosted surfaces), so overlapping glass does not double-tint. This is the spec's correctness mechanism, not an optimization.

### Staleness & repaint policy — the immediate-mode hard part (now designed, was open-question #3)

egui's repaint scheduling makes the browser's "free correctness" model unavailable, and this must be designed, not deferred:

- **No region repaint.** The finest unit egui can repaint is an entire *viewport*; maintainer emilk: "there is nothing built-in for only rendering part of the screen, as all current egui backends assumes a fast hardware rasterizer." Any per-region reuse must be hand-built in a custom backend (hash `ClippedShape`/`ClippedMesh` batches per clip-rect or per 64×64 grid cell) ([emilk/egui#913](https://github.com/emilk/egui/discussions/913)).
- **Per-viewport, smallest-delay-wins schedule.** `request_repaint_after(d)` honors only "the smallest duration" across the viewport; calling it with 1 s every frame still produced ~144 fps because any per-frame immediate request anywhere in the viewport overrides it ([emilk/egui#3109](https://github.com/emilk/egui/issues/3109), [emilk/egui#3931](https://github.com/emilk/egui/issues/3931)). A frosted surface therefore *cannot* rate-limit its own re-grab while anything else in the viewport animates.
- **Reactive by default.** egui repaints only on input/animation; "the UI can become out-of-date if something it is supposed to monitor changes... you need to call `request_repaint()`" ([emilk/egui#4721](https://github.com/emilk/egui/discussions/4721), [docs.rs/egui Context](https://docs.rs/egui/latest/egui/struct.Context.html)). A grabbed backdrop goes stale silently the instant behind-surface content changes without a repaint trigger.

The browser contrast makes the cost explicit: browsers re-snapshot the backdrop *every composited frame* (correctness is free, cost is the constraint — "repaints the backdrop on every scroll frame"), which is why their own guidance is to use `backdrop-filter` on *static* surfaces (modals, navbars), not lists ([MDN backdrop-filter](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/backdrop-filter)). That discipline maps directly onto egui's cheapest correct option.

**The contract:** the Backdrop Root exposes a per-surface **liveness flag** ("is content behind this surface animating?") that selects a repaint policy. There is no automatic middle path, because egui offers no region-level dirty tracking:

1. **STATIC** — grab once when the surface opens, hold. Correct only while behind-surface content does not change; cheapest. The default for tooltips, dialog scrims, navbars over still content.
2. **LIVE** — call `request_repaint()` each visible frame to re-grab every frame. Always correct; pays full-viewport repaint **plus** the grab-pass (~0.2–0.5 ms) continuously, *on top of* a frame the app would otherwise have skipped in reactive mode.
3. **BOUNDED** — `request_repaint_after(period)` for a coarse cadence; valid **only** if the Backdrop Root can guarantee no other widget in the viewport requests faster (rarely guaranteeable per #3109).

The practical shape for a frosted dialog: a brief continuous-repaint window while the surface (and its enter animation) is settling, then a held STATIC grab once both surface and backdrop are static.

### Allocation policy (open question, name the cost)

Bevy always has an offscreen `ViewTarget`; a 2D UI toolkit should engage the intermediate target **only when a frosted surface is on screen** to avoid an unconditional full-screen blit + bandwidth cost every frame. With the direct `egui-wgpu` loop this is straightforward — the loop chooses the color attachment per frame — but the "only-when-needed" path is a design lever, not a settled answer, and switching targets mid-frame is not something eframe's painter supports (another reason the direct loop is required).

### Per-backend strategy

- **wgpu (own loop):** offscreen `Rgba16Float` (or sRGB-typed `Rgba8UnormSrgb`) intermediate via `egui-wgpu::Renderer::render` into a caller-owned attachment → dual-Kawase down/up ping-pong into a mip chain → masked tinted composite (premultiplied linear, §4). Extract the pass from the Bevy/`soorat` pattern; no general crate exists to depend on.
- **glow/GLES (own loop):** `glBlitFramebuffer` grab (GLES3) or `glCopyTexSubImage2D` (GLES2 fallback) → separable/Kawase ping-pong FBO → composite. **Mandatory runtime guards:** query `GL_SAMPLES` (resolve if >0), query GL version (fallback if <GLES3), GLES3 shaders with `#version 300 es` + default precision.
- **Software:** 3-pass box-blur Gaussian approximation (Skia's `window = floor(sigma·3·√(2π)/4 + 0.5)`); box quality is acceptable for the CPU-fallback branch only — **but verify the framebuffer-readback path and measure CPU cost before relying on it** (§3, §8).
- **Web:** same WebGL/WebGPU grab-pass as native if the panel is in-canvas; CSS `backdrop-filter` only if the panel is a separate DOM overlay.

### Algorithm default

Dual-Kawase (ARM 5-tap down / 8-tap up) with iteration count as the radius dial, blurred in **linear light with premultiplied alpha**, downsampling before blur, with optional perceptual-space dithered noise for banding. Linear-sampled separable Gaussian for a single small fixed radius. Pick a consistent half-pixel offset convention and do not mix picom's and KWin's.

### How it degrades

1. **Best:** in-app GPU dual-Kawase (any backend you own) — works everywhere, per-surface, the kiosk answer.
2. **Desktop nicety (fragile):** Route B OS vibrancy via `window-vibrancy` (**macOS `underWindowBackground` only** — drop named materials; **Windows likely blocked for a wgpu app** by the opaque-swapchain occlusion, §2/§3) or `ext-background-effect-v1` (KWin/COSMIC/niri) — *whole-window only*, never per-surface, never under cage.
3. **Software fallback:** CPU box-blur on backends without a GPU (grab path to be verified).
4. **Graceful no-op:** a flat tinted semi-transparent fill (the prior "flat/safe" Glass token) when blur is unavailable or budget-constrained — the degraded glass that still reads as a surface.

### Dependencies (port, don't bind)

Port the dual-Kawase shader (~20 lines, fixed offsets/weights — not copyrightable; re-implement from the published ARM/scenefx values rather than copying picom's GPLv3 text). Do **not** bind Skia (`skia-safe`) or libplacebo (`libplacebo-rs`) just for blur — each drags a heavyweight engine and its own GPU abstraction, an adapter boundary against your wgpu/glow device, justified only if you already render through it. scenefx (MIT, GLES2) is the friendliest *reference* for license-sensitive GLES2 shader text; it is a dependency only if building a wlroots compositor.

## 8. Open Questions / Further Research

1. **GLES/embedded performance (PARTIAL — unmeasured; gates the executive performance verdict).** The sub-ms blur numbers are desktop (GTX 1060, Mali-T760 2015). No 2026-era figures exist for the weak integrated/embedded GPU a cage kiosk likely runs. **Action: in-repo benchmark of dual-Kawase vs separable Gaussian on the actual kiosk hardware for tooltip/dialog-sized surfaces.** Does `Rgba16Float`/sRGB-typed intermediate support hold on that GLES backend?

2. **Premultiplied-alpha at soft glass edges (PARTIAL — under-documented; default chosen but unverified).** The report defaults to premultiplied linear-space blur (matching compositor convention); this is a *decision*, not a measured result. **Action: dedicated de-risk probe** — render a glass panel over a high-contrast edge, compare straight vs premultiplied linear-space blur against a committed snapshot, confirm no dark fringe / white halo. Also: does the W3C spec's "filters operate in sRGB" tolerate the gamma error because box-blur+downsample masks it, or should the toolkit do the more-correct linear blur regardless?

3. **Software-backend grab + cost (PARTIAL — unverified).** No verified path for reading the rendered pixel buffer out of egui's / tiny-skia's software backend, and no cited CPU-blur frame-time figure. **Action: confirm a readback path exists and measure a SIMD box-blur at tooltip/dialog size on the target CPU** before claiming the software branch works.

4. **Intermediate-target allocation cost.** Engaging an offscreen full-screen target only when a frosted surface exists vs unconditionally — the bandwidth/extra-blit tradeoff for a 2D UI toolkit that does not want a Bevy-style always-on `ViewTarget`. Tractable with the direct `egui-wgpu` loop, but the switching policy is unmeasured.

5. **Continuous radius API.** No closed-form sigma↔iterations formula exists in primary sources; KWin interpolates per-pass sampling-distance offsets for a discrete "blur strength." A continuous radius API must interpolate offsets, not rely on a published equation.

6. **Half-pixel offset convention (low — inferred).** The picom-vs-KWin "double-apply" warning was verified against the KWin upsample shader directly but not against picom's current `gl_common.c` side by side. **Action: read both current sources once before porting**, and pick one convention.

7. **Banding amplitude.** The right dithered-noise amplitude/curve for 8-bit vs 10-bit output, and its interaction with the linear-space requirement, is an implementation detail not fully specified.

8. **Route B Windows escape hatch.** Whether a `DxgiFromVisual` DirectComposition swapchain (the only wgpu surface kind that supports transparency) can be wired into an egui/wgpu app to make Mica/Acrylic show — currently blocked by eframe's surface creation and by wgpu's 2025 DirectComposition alpha bug ([gfx-rs/wgpu#7108](https://github.com/gfx-rs/wgpu/issues/7108)). Low priority for the kiosk target (Route B is moot under cage) but the only path to Route B on a wgpu desktop build.

9. **macOS Liquid Glass as a separate capability.** `NSGlassEffectView` is content-glass, not backdrop blur, and its Rust API status is contested (public class per Apple/Microsoft docs; treated as private with App-Store risk by `tauri-plugin-liquid-glass`). If a *content*-glass look is wanted on macOS desktop builds (distinct from backdrop blur), this needs its own evaluation; it does not affect the in-app GPU route or the kiosk.

10. **GPUI Linux/Wayland blur.** Whether GPUI's `WindowBackgroundAppearance::Blurred` gains a real Wayland path (it currently has only X11 xprops) is worth tracking as prior art for the typed API shape — but it offers nothing for cage regardless.

11. **Tiler tint optimization.** Whether `EXT_shader_framebuffer_fetch`'s tile-local read (or `VK_QCOM_tile_shading`'s apron on Adreno) is worth exploiting for the cheap *final tint/composite* step on tilers, even though the blur itself still needs the grab + ping-pong.

12. **Stale-backdrop invalidation beyond liveness flag.** §7 designs a per-surface liveness policy; a more granular region-invalidation scheme would require a custom egui backend that hashes clip-batches (emilk's DIY sketch). Whether that is worth building for a kiosk with few frosted surfaces is unresolved.

13. **cage patchability.** Whether a small patch adding `ext-background-effect-v1` or scenefx to cage is feasible — which would change the kiosk recommendation, but is currently nonexistent upstream and would still be semantically moot for a single fullscreen app.

## Verification & Confidence

**Adversarially CONFIRMED (load-bearing wins, sound under scrutiny):**
- The wgpu blocker is the surface's `RENDER_ATTACHMENT`-only guarantee plus the toolkit's single clearing pass — **not** an inherent wgpu limit (verbatim docs).
- Bevy's bloom node samples the offscreen `main_texture_view`, not the swapchain — the production existence proof for grab→blur→composite in wgpu.
- **The egui-specific seam is real and fork-free:** `egui-wgpu::Renderer::render` takes a caller-chosen color attachment; the whole UI flows through it; eframe's painter is what blocks the clean route, not egui-wgpu (this closes the prior draft's single biggest unproven assumption).
- The cage blur **REFUTATION**: cage's source creates ~21 wlroots globals, none a blur manager; the wayland.app "Cage 0.2.0" entry is a genuine scraper false-positive (corroborated by the same table listing Sway/river/Weston, which upstream-refuse blur).
- Vello's `draw_blurred_rounded_rect` is a shape primitive; general blur (#476) is unimplemented and not a backdrop filter.
- Dual-Kawase 5-down/8-up tap pattern matches current KWin master and picom; relative performance ordering and bandwidth rationale are durable.
- Gamma: blur must be in linear light (opaque content).
- **egui's repaint model**: per-viewport, smallest-delay-wins, reactive by default, no region invalidation — the immediate-mode staleness problem is real and now has a designed liveness-keyed repaint contract (§7).

**DISPUTED → corrected in this revision:**
- **GPUI "macOS-only" was over-corrected to REFUTED in the draft.** Correct status: Windows DWM blur **added** (CONFIRMED); Linux blur **UNCONFIRMED** (X11-xprop only, no Wayland path, useless under cage).
- **Route B Windows for a wgpu app was marked CONFIRMED-high; it is likely blocked by default.** wgpu's default HWND swapchain "does not support transparency" (rustdoc verbatim); `PreMultiplied`/`PostMultiplied` exist only on DirectComposition surfaces (wgpu#7117); an opaque present occludes the DWM backdrop; eframe Windows transparency is itself open (#4451). Tauri's WebView2 topology is not evidence for an egui/wgpu app.
- **macOS Route B ignored macOS Tahoe 26.** Correction: the `underWindowBackground` behind-window blur primitive **survives** and remains the only public path; the **named decorative materials** (Sidebar, HudWindow) are superseded by automatic Liquid Glass and should be dropped; `NSGlassEffectView`/Liquid Glass is content-glass, not backdrop blur, with a contested public/private API status in the Rust ecosystem.

**Remaining UNCERTAIN (carried to Open Questions, not papered over):**
- Performance on the actual weak-GLES kiosk GPU (no 2026 figures) — the executive performance leg is **PARTIAL**, not CONFIRMED.
- Premultiplied-alpha at the translucent rounded-rect edge — a default is chosen (premultiplied linear) but **unverified**; needs a snapshot probe.
- Software-backend framebuffer-readback path and CPU cost — **unverified**; no readback path or frame-time figure located.
- The picom-vs-KWin half-pixel double-apply failure mode — **inferred**, not confirmed by a side-by-side source read.
- The "engage intermediate target only when needed" allocation policy — viable in principle with the direct `egui-wgpu` loop, but unmeasured.

**Net:** the in-app GPU backdrop-blur core is well-verified and publishable, and the project-specific implementation path is now concrete (drive `egui-wgpu`/`egui-glow` directly; do not use eframe's painter; own a liveness-keyed repaint policy). The corrections concentrate in the OS-vibrancy fallback (Route B is weaker on both Windows-wgpu and macOS-Tahoe than the draft claimed) and in three honestly-unresolved measurement gaps (kiosk-GPU performance, translucent-edge gamma, software-backend grab) — none of which undermines the central thesis that in-app GPU grab→blur→composite is the correct, and for cage the *only*, route.
