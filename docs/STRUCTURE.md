# `backdrop-blur` — Recommended Crate Structure

> **SUPERSEDED IN PART (2026-06-05): Iced was dropped as a target** — desktop-focused and its
> render API can't cleanly do backdrop blur (see `DESIGN.md` §7). Treat every `backdrop-blur-iced`
> crate / `iced-panel` example / `iced` feature below as **not planned** — they stand only as the
> generic *"one crate per toolkit adapter"* pattern, with **egui the sole planned adapter** and any
> future toolkit an additive crate. Mentions of *iced as an ecosystem precedent* (how iced
> structures its own workspace) are retained — that is the verified evidence the topology rests on.

## 1. Verdict

**Build a cargo workspace of multiple published crates — a pure backend-agnostic core, one crate per GPU backend, one crate per toolkit adapter, and an optional thin facade — not a single feature-gated crate.** This is the unanimous topology of the two closest precedents: **egui** ships its core (`egui`/`epaint`, zero GPU/winit deps) with `egui_glow` and `egui-wgpu` as *separate published crates* re-exported behind `eframe`'s additive features (confirmed against the live egui source), and **iced** ships `iced_core`/`iced_wgpu`/`iced_tiny_skia`/`iced_renderer` as separate workspace members with the facade feature-gating the backend (confirmed against iced master). The decision is forced, not stylistic: backdrop-blur's public seam re-exports backend resource types (`wgpu::TextureView` vs `glow::Texture`) which are mutually-distinct and which make wgpu/glow *public dependencies* — and a stable crate cannot be capped at two pre-1.0 backends churning in lockstep behind one version number. The crate split isolates each unstable public dependency so a wgpu major bump majors only `backdrop-blur-wgpu`.

## 2. The Hard Constraint

**The rule (citation):** The Cargo Book, *Features*, is normative: *"Features should be additive. That is, enabling a feature should not disable functionality, and it should usually be safe to enable any combination of features."* Cargo enforces this by **feature unification** — *"When a dependency is used by multiple packages, Cargo will use the union of all features enabled on that dependency when building it."* On exclusivity it states mutually-incompatible features *"should be avoided if at all possible, because it requires coordinating all uses of the package in the dependency graph,"* and lists the first remedy as *"Split the functionality into separate packages."* (https://doc.rust-lang.org/cargo/reference/features.html)

**The consequence for backdrop-blur:** Whether `wgpu` and `glow` *can* be two features of one crate hinges on one fact — **do their code paths compile together cleanly?** They cannot share one public seam, because the seam's resource types differ (`wgpu::TextureView` vs `glow::Texture`); a single struct API cannot serve both. So a single backend crate would either (a) force one module/public-type to serve two irreconcilable resource universes, or (b) model the choice as two features one of which disables the other — both of which violate the additive rule. Under feature unification, two downstream consumers each picking a different backend feature would union *both* backends into one build. **Therefore the backends MUST be separate crates** ("which backend" = "which crate you depend on," not "which feature you flip"). This is exactly why egui ships `egui_glow` and `egui-wgpu` as distinct crates rather than `egui --features glow|wgpu` (verified: both are independently published at 0.34.3, not modules).

**Where features remain legal:** Features are correct only on the *facade*, where each feature pulls a *different optional sub-crate* (`egui = ["dep:backdrop-blur-egui"]`) — additive, never mutually-exclusive, exactly the eframe pattern. Cargo's additive rule is satisfied because enabling both `wgpu` and `egui` simply compiles two more optional crates; neither disables the other.

## 3. Proposed Layout

The core is a **types/seam crate that backend crates implement** — *not* a core generic over a `Backend` parameter, and *not* feature-gated. The seam is a **static-dispatch associated-type trait** (the `wgpu-hal::Api` model: a trait carrying the backend's resource types as associated types, deliberately *not* object-safe), living in core. Backend crates define a local marker type and `impl BackdropBlur for WgpuBlur`. The orphan rule makes this legal (trait is in the depended-upon core, the impl type is local) and *favors* it; it is exactly the `wgpu-types` (pure vocabulary, no backend deps — confirmed: depends only on bitflags/bytemuck/log/raw-window-handle) → `wgpu-hal` (backend impls) layering, scaled down. The seam is *not* object-safe (`raw-window-handle`'s traits are object-safe because handles are passed as `&dyn`; here resource types must differ per backend, so associated types + monomorphization + static dispatch is the right call, per API Guidelines C-OBJECT — make object-safe *only if* trait-object use is wanted, which it is not).

```
backdrop-blur/                         (NEW dedicated repo; virtual-manifest workspace)
├── Cargo.toml                         [workspace] only — no [package]; resolver = "3"
├── crates/
│   ├── backdrop-blur-core/            core vocabulary + the seam trait; ZERO GPU/toolkit deps
│   ├── backdrop-blur-wgpu/            wgpu impl of the seam; carries WGSL shaders
│   ├── backdrop-blur-glow/            glow impl of the seam; carries GLSL/GLES shaders (the unsafe one)
│   ├── backdrop-blur-egui/            egui adapter: glow grab-pass + wgpu own-loop paths  (sole planned adapter)
│   └── backdrop-blur/                 OPTIONAL thin facade; re-exports behind additive features
└── examples/
    ├── egui-glow-panel/               own crate → deps backdrop-blur-egui (glow) + eframe
    └── egui-wgpu-panel/               own crate → deps backdrop-blur-egui (wgpu) + eframe
```
*(A `backdrop-blur-<toolkit>` adapter crate + `<toolkit>-panel` example is the additive slot for any
future toolkit that exposes a sampleable-backdrop hook; none is planned — iced was dropped, §DESIGN 7.)*

**Per-crate responsibility + dependencies (one line each):**

| Crate | Owns | Depends on |
|---|---|---|
| `backdrop-blur-core` | `BlurParams` (Kawase iterations/offset, linear-space flag), `Region`, `RoundedRectMask`, `Tint`, the shader-source description, and the `BackdropBlur` associated-type seam trait — all pure, headless-testable, `#![forbid(unsafe_code)]` | nothing GPU/toolkit (maybe `bytemuck`) |
| `backdrop-blur-wgpu` | `WgpuBlur` marker + `impl BackdropBlur` over `wgpu::TextureView`/`wgpu::Device`; WGSL | `core` + `wgpu` |
| `backdrop-blur-glow` | `GlowBlur` marker + `impl BackdropBlur` over `glow::Texture`; GLSL/GLES; the `unsafe` GL FFI confined here | `core` + `glow` |
| `backdrop-blur-egui` | egui integration: the glow grab-pass path *and* the wgpu own-loop path, each a sub-feature | `core` + `egui`(opt) + `-glow`(opt) + `-wgpu`(opt) |
| `backdrop-blur` (facade) | one-import convenience: `pub use` core + chosen sub-crates behind additive features | optional deps on all of the above |

*(The dropped `backdrop-blur-iced` row — `iced custom-shader-Primitive integration`, `core` + `iced_wgpu` + `-wgpu` — is the template a future toolkit adapter would follow; not planned, §DESIGN 7.)*

**Dependency DAG (strict, core at the sink, no sibling knows another):**

```
backdrop-blur (facade, optional)
        │  feature-gated optional deps
        ▼
 backdrop-blur-egui   (sole planned adapter; a future toolkit adapter slots in beside it)
        │   │
        │   └──► backdrop-blur-wgpu
        └──────► backdrop-blur-glow
                       │   │
                       ▼   ▼
                 backdrop-blur-core   ◄── wgpu/glow crates also sit at this level
```

This passes the "design for the second feature" test: a third backend or third toolkit is a *new crate against the stable core trait*, never a core rewrite — the way glyphon could add glow later without touching cosmic-text.

## 4. Adapter crates — each toolkit is its own crate (egui is the only planned one)

**Each toolkit adapter is a separate published crate (`backdrop-blur-egui`; a future toolkit would be its own `backdrop-blur-<toolkit>`), not a feature of one crate.** Cited precedent: egui keeps integration code *out* of the core — `egui_glow` and `egui-wgpu` are independently published, independently versioned crates (confirmed live), and `bevy_egui` (the canonical `*-egui` third-party adapter) versions on its *own* cadence (0.40-rc against bevy 0.19 + egui 0.34), decoupled from both hosts. The blast-radius argument is concrete and verified: `egui-wgpu` repeatedly broke on wgpu major bumps (0.20→22→25; issues #4850/#4905/#5476) and could re-release against each bump *without* forcing a version churn on the egui core or on glow-only consumers. The tracing ecosystem corroborates beyond GUI (facade `tracing` + separate `tracing-opentelemetry`/`axum-tracing-opentelemetry` integration crates). The hard requirement this satisfies: **a glow-only user must never compile wgpu, and a future non-egui adapter's user must never compile egui's dep tree.**

Within `backdrop-blur-egui`, the two egui render paths are *additive sub-features* (`grab-pass` → pulls `-glow`; `own-loop` → pulls `-wgpu`), mirroring how `egui-wgpu`/`egui_glow` gate their optional `winit` integration behind a `winit` feature — a consumer pulls only the path it renders with.

## 5. Feature Flags

Names are bare and additive per API Guidelines **C-FEATURE** (*"Name the feature `abc` directly,"* never `with-wgpu`/`use-glow`). Use `dep:`/weak-`?/` syntax (MSRV ≥ 1.60).

```toml
# backdrop-blur-core/Cargo.toml
[features]
default = []
serde = ["dep:serde"]          # additive; gates derives on BlurParams/Region/Tint

# backdrop-blur-wgpu/Cargo.toml
[features]
default = []
serde = ["dep:serde", "backdrop-blur-core/serde", "wgpu?/serde"]

# backdrop-blur-glow/Cargo.toml
[features]
default = []                   # safe core deps only; the unsafe GL is unconditional code, not a feature

# backdrop-blur-egui/Cargo.toml
[features]
default = []                   # consumer picks a path explicitly
grab-pass = ["dep:backdrop-blur-glow"]   # egui glow grab-pass
own-loop  = ["dep:backdrop-blur-wgpu"]   # egui wgpu own-loop
egui      = ["dep:egui"]

# backdrop-blur-iced/Cargo.toml
[features]
default = ["wgpu"]
wgpu = ["dep:backdrop-blur-wgpu"]        # iced's only path today is wgpu

# backdrop-blur/Cargo.toml  (facade — features switch whole sub-crates, eframe/iced model)
[features]
default = []                   # ← conservative: empty default; consumer opts into what it ships
wgpu = ["dep:backdrop-blur-wgpu"]
glow = ["dep:backdrop-blur-glow"]
egui = ["dep:backdrop-blur-egui"]
iced = ["dep:backdrop-blur-iced"]
```

**Default features — conservative, deliberately.** SemVer makes the default set a one-way door: *"It is usually safe to add new Cargo features"* (MINOR) but removing a feature, or removing one from `default`, is a *breaking* (MAJOR) change (https://doc.rust-lang.org/cargo/reference/semver.html). The facade and core therefore default to **empty/core-only**; adding a backend or toolkit later is a free minor bump, dropping one is impossible without a major. This diverges from eframe (which defaults `wgpu`-on) on purpose — backdrop-blur cannot guarantee both backends compile on every target forever, so it never default-enables a backend.

**docs.rs:** every published crate gets, per the image-crate touchstone:
```toml
[package.metadata.docs.rs]
all-features = true
rustdoc-args = ["--cfg", "docsrs"]
```
so the feature-gated seam (wgpu + glow + adapter surfaces) renders fully despite the empty default set.

## 6. Naming, Edition/MSRV, Unsafe, Examples, Repo

- **Crate naming:** hyphenated `backdrop-blur-{core,wgpu,glow,egui,iced}` + facade `backdrop-blur`, so the toolkit×backend matrix is legible from the crate name alone (the `egui-wgpu`/`<core>-<toolkit>` convention). Core is named for *capability + role* (the `wgpu-types` analog), not implementation — per the project's "domain names, never `Manager`/`Handler`" rule. Type names follow **C-CASE**: `Wgsl`/`Gles`, not `WGSL`/`GLES`.
- **Edition / MSRV:** `edition = "2024"` (matches the host `abdu-egui-ui` repo and current iced). **`edition = "2024"` *requires* Rust ≥ 1.85.0**, so `rust-version` is pinned to **`"1.92"`** = max(1.85 edition floor, 1.87 wgpu 29, 1.92 egui/egui-wgpu 0.34) — the egui 0.34 MSRV is the binding constraint, not the edition floor. The earlier "≥ 1.64 / weak-dep needs ≥ 1.60" justification was a contradiction (a 1.64 MSRV on an edition-2024 crate fails to build on 1.64–1.84) and is dropped — edition 2024 subsumes the `dep:`/weak-dep and workspace-inheritance floors. A CI job pins the toolchain to the declared MSRV so it cannot silently drift. `license = "MIT OR Apache-2.0"` (**C-PERMISSIVE**, matches the host repo); populate description/repository/keywords/categories (**C-METADATA**).
- **Unsafe policy — isolate the unsafety to the one crate that cannot avoid it.** `backdrop-blur-core` is `#![forbid(unsafe_code)]` (it is pure math + types + a trait — the safety guarantee is part of its contract and machine-checked). `backdrop-blur-wgpu` should also be `#![forbid(unsafe_code)]` (wgpu's API is safe). `backdrop-blur-glow` is the *only* crate where `unsafe` lives — raw GL FFI through `glow` is unavoidable. Confine it there with `#![deny(unsafe_op_in_unsafe_fn)]`, a `// SAFETY:` comment on every block (per the project's unsafe rule), and **per Abdu's standing rule, the `unsafe` GL blocks require his approval before they're written.** Adapters depending only on the wgpu path inherit `forbid(unsafe_code)`; the structure makes "did I pull in unsafe code?" answerable by "did I depend on `-glow`?".
- **Examples:** each per-adapter demo (frosted panel over moving content) is its *own crate* under `examples/` (the iced `cargo run --package <example>` convention; vello does likewise "for independent dependencies and faster builds"). This keeps eframe/winit/iced *out* of every published library's dependency graph — the direct analog of how the host repo gates its eframe preview behind a `preview` feature to keep the default loop portable. Never `[[example]]` targets inside the library crates.
- **Repo:** a **new dedicated repository**, not a member of `abdu-egui-ui`. Every precedent at this scale (wgpu, vello, egui, iced) is its own repo. backdrop-blur is rendering-backend-agnostic and must not be chained to the egui UI lib's release cadence; `backdrop-blur-egui` is merely *one consumer* of it.

**Workspace mechanics:** root is a **virtual manifest** (no `[package]`; the egui/wgpu/vello model — note iced's root is a *real* package, a deliberate outlier I do not follow), `resolver = "3"` set explicitly, shared wgpu/glow/egui/iced versions pinned once in `[workspace.dependencies]` (entries can't be `optional` — members set `optional = true` locally), and version/edition/rust-version/license inherited via `[workspace.package]` so published crates move in lockstep. One `Cargo.lock`, one `target/`. Each published crate ships its own `README.md` + `CHANGELOG.md` + `LICENSE-MIT` + `LICENSE-APACHE` (the Linebender/vello per-crate convention). CI runs per-crate `--all-features` + builds each example crate; **no** `cargo-hack --feature-powerset` is needed because the backend split is crate-level, so there is no exploding intra-crate feature matrix.

## 7. Minimal v1 vs Full (YAGNI)

The full topology above is the *target shape* and what the workspace leaves room for. **Do not create all seven crates on day one.** The crate split is justified only where a real, heavy, distinct dependency or a forked public type surface crosses the boundary; the community guidance is explicit that *"a crate with fewer than 500 lines and only one dependent probably shouldn't be its own crate."* Build the smallest correct topology that still proves the seam and ships one visible result:

**v1 — create these four crates now:**
1. `backdrop-blur-core` — the seam trait + params/region/mask/tint vocabulary. This is the load-bearing design artifact; get it right first. (Solves the hard problem: does one trait abstract both backends without an adapter tax?)
2. `backdrop-blur-wgpu` — the first backend. wgpu is the lower-risk, safe-API backend that the egui own-loop path needs, so it unlocks v1 with zero unsafe.
3. `backdrop-blur-egui` (own-loop/wgpu path only) — the first adapter, proving the seam composites into a real toolkit.
4. `examples/egui-wgpu-panel` — the visual proof.

**Deferred until there's pressure (the structure already accommodates each as an additive new crate, no rewrite):**
- `backdrop-blur-glow` + the egui `grab-pass` sub-feature — adds the *only* unsafe crate and a second resource universe; defer until a glow-only deployment actually needs it (and until the `unsafe` blocks are approved). The seam trait having *proven* it abstracts wgpu is the precondition for adding glow cheaply.
- ~~`backdrop-blur-iced`~~ — **dropped** (desktop-focused + `Primitive::render` can't sample a backdrop; DESIGN §7). The toolkit-adapter slot stays open for any *future* toolkit that exposes a sampleable-backdrop hook — an additive new crate against the stable core, the "second feature" test — but none is planned.
- `backdrop-blur` facade — add *only* when ≥ 2 sub-crates exist and a one-import entry point earns its keep (wgpu/vello/egui mostly *don't* ship a facade; iced and eframe do). A facade that has to gate between backends would re-import the mutually-exclusive-feature problem the split exists to kill, so it must stay additive-re-export-only or not exist.

The single non-negotiable that must be right in v1 because everything hangs off it: **the `BackdropBlur` seam trait in core.** Get the associated-type shape correct (backend resource types in the slots, region/params/tint as concrete value types, static dispatch, not object-safe) and the rest is additive crates; get it wrong and adding glow forces a core rewrite.
